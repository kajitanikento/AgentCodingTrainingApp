# Agentへの実装指示

## プロジェクト構成の注意事項

### PBXFileSystemSynchronizedRootGroup
このプロジェクトは `PBXFileSystemSynchronizedRootGroup` を使用している。
ファイルシステム上に存在するファイルは自動的にXcodeプロジェクトに認識されるため、`.xcodeproj` へのファイル追加は不要。
LSPのキャッシュが古い場合でも、ファイルが存在していれば正しくビルドされる。LSPの補完・エラー表示を過信せず、実際のファイル存在を優先して判断すること。

---

## 実装前に必ずやること

### 1. 設計書の確認
- アーキテクチャルールを設計書で確認してから実装を開始する
- ファイルの置き場所・クラスの責務が設計書と一致しているかを確認する

### 2. 既存コードの調査
- 実装しようとしている機能・コンポーネントに類似した既存実装がないか検索する
- UIコンポーネントとロジック・ユーティリティをそれぞれ調査する
- 既存のものが使えるなら新規作成せずに流用する

---

## 実装時のルール

### テストコード
- 受け入れ条件をすべてテストケースに落とし込む
- 各テストのdocコメントには「何をテストするか」と「期待値」を自然言語で明記する
- assertionは期待値と正確に対応させる（`XCTAssertNotNil`のような曖昧なassertionは使わない）

```swift
/// 未ログイン状態でAPIを呼び出したとき認証エラーが返る
/// 期待値: AuthError.unauthorizedが返る
func testUnauthenticatedRequestReturnsUnauthorizedError() {
    let result = sut.call()
    XCTAssertEqual(result, .failure(.unauthorized))
}
```

### セキュリティ
- 外部入力は必ずバリデーションする
- 機密データ（トークン・パスワード等）はKeychainに保存する（UserDefaultsは使わない）
- ログに個人情報・認証情報を出力しない

## PRを作成するときにやること

### 1. PRテンプレートをすべて埋める
`.github/PULL_REQUEST_TEMPLATE.md` の全セクションを埋める。

### 2. 設計判断を記載する
新規追加したファイル・クラスについて、なぜその場所・構造にしたかを設計書を引用しながら記載する。

### 3. 重複チェック結果を記載する
調査したキーワードと結果（既存流用 or 新規作成の理由）を記載する。

### 4. セキュリティチェックリストを確認する
該当する項目がある場合はファイル名・行番号も記載する。
